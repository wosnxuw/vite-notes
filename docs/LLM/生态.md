## Skills

现在这个 Claude 把 Skills 的标准公开了，很多工具现在也支持

https://code.claude.com/docs/en/skills

元数据：必定加载（目录），用于降低工具太多导致的 Token 消耗

指令：正文

资源：其它文件。可以是放在 SKILL.md 旁边的 markdown，也可以是放在子文件夹的脚本，子文件夹、文件如何命名都可以，只要在 SKILL.md 说明，并且让 AI 能找到即可。

每个 skill 是一个文件夹（对比 mcp，它是一个可执行程序），包含一个 SKILL.md，此 SKILL.md 包含了**元数据**和**指令**。即指令是直接放在元数据下面的。

然后罗列一大堆子文件夹，每个子文件夹都是资源，资源可以是图片，文本，也可以是脚本

元数据：是 Markdown 的元数据，和我用 hugo 的时候必须填的那个头类似，一个 yaml

```markdown
---
name: explain-code
description: Explains code with visual diagrams and analogies. Use when explaining how code works, teaching about a codebase, or when the user asks "how does this work?"
---

When explaining code, always include:

1. **Start with an analogy**: Compare the code to something from everyday life
2. **Draw a diagram**: Use ASCII art to show the flow, structure, or relationships

- For complete API details, see [reference.md](reference.md)

```

指令：就是写一个特别详细的提示词，来提示 AI 如何操作

资源：如果不需要脚本，有时不需要子文件夹里的资源

注意，每次你提问，Claude 会扫描你所有的 SKILL 的元数据，就和工具类似，你只要开启，就直接加入上下文，所以你不用，就不要开。

SKILL 的存放位置：

全局：`~/.claude/skills/<skill-name>/SKILL.md`	所有的项目

项目：`.claude/skills/<skill-name>/SKILL.md`	仅限此项目

一句话：Skills 是一种提示词管理模式，相当于大号提示词数据库，并支持渐进式的加载提示词，避免上下文太长。MCP 是一个工具箱，你不要一次给太多，否则 AI 也有选择困难症。

我认为直接抄别人的 SKILL 是比较好的，一般不要自己写。SKILL 有一点类似于 Bash 即一切，通过 Bash 来运行命令，或者运行已有脚本，来操作更多的东西。对比 MCP 的话，MCP 可以更精准的做某些事情，比如在 FreeCAD 里执行操作（但是其实通过 SKILL 里提供一个 python 脚本来要求 AI 运行，应该也能有类似效果）

### n8n

[2026-1-19]

这东西去年就开始火，但是我一直没用，今天试了一下

这东西号称是 no-code，但是实际上，所有的东西都是微服务设计，架构分离，该少的东西你一个不能少（比如你要搭 RAG，那么你一样要部署 Milvus），但是灵活性是够的

## Vibe Coding

我对比了一下 Vibe Coding 的工具，分为如下几类：

（1）VSCode 本身 + 各种插件

各种插件毕竟是第三方的，可能 UI 不是那么美好（其实就是内嵌一个网页，高分辨率下，很多字体大小都不统一）

（2）VSCode 分叉版

Cursor 刚火的时候试过。我看 Cursor 应该是有点黑科技的，尤其是读他们的博客。应该是内置了很编译器数据库，据说是先跑小模型理解你的意愿，再找内容，输入给 AI 要看的内容很准，也省 Token。
Cursor 本身也是调 Claude（抛开价格优势），那为什么不买 Claude 会员呢？说明还是有些东西。

Trae 半年前试过。

Antigravity Google 出的，目标是多 Agent 同时协作。我没下载。我看 reddit 上的评价不是特别好？目标群体是那种顶级 AI Boy，依靠十几个 AI 协作，每天上万行代码，甚至无法人工 review 的极客。

（4）Zed

应该是没有 Cursor 那种黑科技的，但是 AI 确实内置到了侧栏，UI 不赖。和 IDE 内置的类似，也是通过查看文件的方式来工作

（5）命令行

比如 Claude Code、Codex、Qwen Code、Gemini CLI、Open Code

这类工具的思路是，给 AI 提供 MCP 工具，让 AI 真正的像人类一样，先 ls 查看项目结构，然后再 grep 查看关键点

总结：受限有于训练成本，大模型本身确实是 AI 时代的护城河。但是外围生态有点隐约成为城墙的趋势。现在一看模型答题评分都巨高，但是实际体验确实是差距。

### Claude Code

[2026-3-3]

其实很早就下载了这玩意，毕竟大名鼎鼎了算是。但是因为没订阅 Claude，所以随便给了个 ds 就扔一边了，一直没怎么仔细的用起来。

最近用了一下，我发现它确实有说法。它的提示词系统更加精细、完备。即使同一个模型，换到 Claude Code 上也会表现更好，超过 OpenCode。

根据我的一些了解，做这种 CLI 工具，它的思想其实就是 Bash 操纵一切。

一个最简单的 CC 如下，一个 python 文件，就给模型仅仅一个工具，那就是执行命令，然后返回命令的执行结果

子 Agent 不需要实现（只需要通过提示词告诉 AI 有这么一回事），因为通过在 bash 里重新执行 python_cc.py 即可唤起一个子 agent 工作，然后获取其结果

需要看就 cat，需要写就 echo 之类的

然后里面搞一个死循环，一直调 Agent 工作即可

**重点：**

其实我最近发现上下文管理（提示词系统很有用），如果把大模型比作 y = f(x)，你想要更好的 y，那么确实 x 需要更好

CC 的提示词大概有几个点：

（1）描述何时使用子智能体：读取大量文件，任务是独立的，避免细节污染当前对话

（2）不要做多余的事情

而且 CC 是会做项目的记忆的，把重点记下来，以便于后续查找，而且是分层的。

Claude.md 用于总体管理，并且每个子目录都可以放子的。

而且对话中，还会创建任务清单，通过打勾的方式确认做了什么。

CC 还会像人一样，搜索代码，实时阅读，并不需要把所有东西放在上下文里。修改也是按行修改，省上下文。

只能说它确实有效果。

## Blog

用于总结 Claude Cursor Manus 等等喜欢发博客的公司的博客

https://claude.com/blog/best-practices-for-prompt-engineering

按照输入输出理论，我确实应该提高输入的质量了

总结了下这篇文章的亮点，与 AI 对话时，应该做什么？

（1）直接说明你想做什么。

使用简单的语言，明确表达你的需求，且不含糊。不要绕弯，不要模糊。

（2）提供做事的动机。

不要：不要使用要点

要：我更喜欢以自然段落的形式回应，而不是以要点来表达，因为我觉得流畅的散文更容易阅读，也更善于交谈

（3）告诉人工智能该做什么，而不是不该做什么。

不要：不要在回复中使用“markdown”。

要：你的回复应由流畅流畅的散文段落组成。

（4）将你的提示样式与期望的输出相匹配。

如果你的输入标记很多，那么他也会这样

（5）使用 XML 来提高清晰度似乎没那么大用。

（6）你是数学领域专家。基本没用，过于具体的角色可能会限制人工智能的帮助。
