### 文件描述符 fd

File Discriptor

Linux 进程总是有 0=stdin、1=stdout、2=stderr，但是它们接入到哪里，是这三个文件描述符执行的对象

1、通过 shell 启动其他程序

最常见的情况，首先要 fork 出子进程，此时有一个修改 fd 的机会，重定向和管道可以修改 fd，设置好以后，exec 执行

2、运行时换 fd

通过 dup2() dup3() 可以换 fd 1。

3、伪终端

交互式进程的 fd 通常是设备文件

### 重定向和管道

重定向：命令与文件之间

管道：命令与命令之间

#### 重定向

输出重定向 `>` `>>`

输入重定向 `<`

`>`：将命令的标准输出连接到文件，覆盖/新建 `echo "Hello" > file.txt`

`>>`：同上，但是追加 `echo "Hello" >> file.txt`

`<`：将命令的标准输入换成文件 `sort < file.txt`

1、如果要写入内容，你可以 `echo hello > file`，而不是 `hello > file`，必须有 echo，否则你没有进程的 stdout

2、如果把“文件”理解为“文件系统的一个路径”（有的文件是`/dev/null`）

那么不管是 `>` 还是 `<`，右边都是文件。（没说左边一定是命令，这也变相在说，命令的最开始必须是类似于可执行的东西，而不是文件）

例子：`grep -v '^$' < messy.txt > clean.txt`

### 管道

`|` 把左边的命令的输出连接到右边命令的输入

例子：`ps aux | grep nginx`

#### Heredoc

fish 不支持 `<<`

`<<` 和之前的重定向类似，右边是一个文件，只是这个文件就在此处构造，左边没说

多行语法：

```shell
命令 << 结束标记
多行文本内容1
多行文本内容2
多行文本内容3
结束标记
```

结束标记一般是用 EOF，只是一个标记，并不是真正的关键字，需要单独在一行

从 `cat > file.txt << abcd` 来思考

启动 cat，并设置 cat 的 stdout 是 file.txt，stdin 是 heredoc，即那个字符串

效果和 `cat << abcd > file` 是一样的

虽然从数据流动的方向看，cat 像是一个管道，但是它不能放在中间，否则 shell 不知道启动什么程序

当然你也不能直接 `abcd | file`

如果是管道，应该是

`printf 'hello world' | tee file`

`echo 'hello world' | tee file`

脚本里推荐 printf，更精准

heredoc 也可以直接跑脚本：

```shell
python3 << EOF
print("hello")
EOF
```

### 其它

`||` 逻辑“或”，如果左边失败了才执行右边。多用于兜底输出失败日志

`&&` 逻辑“或”，如果左边成功了才执行右边

`&` 是将命令放在后台执行（别占终端）例子：`tar -xzf big_file.tar.gz &`

备注：除非你就要并发执行，否则不要写 `command1 & command2`，因为 shell 做语法分析后，会并发执行两个任务，终端上跑的是 cmd2

### 命令行参数

命令行参数和标准输入是两码事

命令行参数，只有程序最开始跑的那一刻输入进去了，之后没机会了

标准输入则在整个周期都可以输入，还可以换

前者是 `char *argv[]`，后者是 `FILE *stdin`

即：`echo "hello"` 的时候，是没有 stdin，argv 是 hello，stdout 是 hello

`cat < file.txt` 的时候，是没有 argv，stdin 是 file.txt 的“内容”，stdout 也是同样

`cat file.txt`，是解析到命令行参数里有文件，然后 cat 自己打开那个文件，输出内容

### 常见工具总结

现在混乱的根源是：各种工具对待命令行参数，和标准输入的态度不一致

cat：有 arg，则把 arg 当做文件路径，输出那个文件的内容；无 arg，去看 stdin，复制 stdin 到 stdout

echo：只看 arg，复制到 stout

printf：只看 arg，并且是参数化的 arg，不看 stdin

tee：必须同时有 arg 和 stdin，arg 是要写入的文件路径，stdin 是源文件，复制那个文件两次，一份到 stdout，一份到 arg 指向的位置

### 交互式进程

你不能说一个进程是或者不是交互式，它是进程运行的状态

非交互式：`python main.py`, `bash script.sh`

交互式：`python`, `bash`, `vim`

非交互式就自顾自的运行了，而交互式则相应你键盘的输入（其实它的 stdin 是 tty）

### TTY

Teletypewriter 电传打字机

#### 为什么有

Unix 哲学坚持简单抽象，“任何能产生或消费字节流的东西，都应该以文件的形式呈献给程序员”

TTY 存在的根本原因是解耦，通过“作为文件”，允许应用程序使用简单的系统调用来处理复杂的终端逻辑

否则，应用程序要自己解析键盘硬件

对用户进程（面向程序员）——提供熟悉的 read/write API。Shell 和 Vim 只需写入 /dev/tty 即可显示，无需关心数据去往何处

一个统一的抽象层

#### 主从设计

主端（/dev/ptmx）是内核端，从端（/dev/pts/*）是用户端

将“谁在操纵硬件”（终端模拟器 Ghostty）与“谁在使用终端”（Shell/Vim）分开，两者只通过字符设备文件进行对话

“输入”和“控制”要分开，在图形化界面里解析到 Ctrl+C 时，将 SIGINT 发送到 slave 端

#### 网路即文件

使用 ssh 时，你的图形化终端模拟器本身就有一对 TTY，这里的 本地 PTY 就给 ssh 用（即 ssh 类似 python）

ssh 把包发送给 sshd

因为远程主机的 bash，无法直接工作，也得像你站在屏幕前那样，所以 sshd 替代了 Ghostty 的位置，在远端创建了一对 TTY，其中远端的 PTY 给 bash 用
