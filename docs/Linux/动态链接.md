### 理念

一台机器上，程序尽量共享同一份库，由发行版统一维护、升级和打补丁

第一，省空间、省内存。第二，安全更新集中化。第三，发行版可以做整体集成测试（全系统依赖图）。第四，ABI 稳定时体验很好

这在只用发行版仓库里的软件时很舒服，但最大的缺点是：把“应用依赖”提升成了“系统状态”

程序能不能跑，不只取决于你的代码，还取决于这台机器的 /usr/lib、ld.so.cache、LD_LIBRARY_PATH、包版本、发行版版本、编译器 ABI、glibc 版本等一堆全局状态。

Docker 解决不了所有 ABI 问题，尤其是涉及内核、GPU 驱动、glibc 版本、CUDA、NVIDIA runtime、宿主机设备映射时，因为容器共享宿主机内核

Linux 命令行世界里，“应用”经常不是一个巨大程序，而是一堆小程序的组合生态

### SemVer 版本号

Semantic Versioning 语义化版本
```
MAJOR.MINOR.PATCH
主版本。次版本.补丁版本
```
MAJOR：做了不兼容的 API 修改

MINOR：增加了向后兼容的新功能

PATCH：做了向后兼容的问题修复

问题：实际执行程度不一，很多项目不遵守

### 程序如何找 so

1、`ldd <程序名>` 可以检查依赖关系

2、`export LD_LIBRARY_PATH=/自定义/路径:$LD_LIBRARY_PATH` 调试时，可以新增一个路径

3、固化在二进制程序内部的路径，DT_RPATH（古老）DT_RUNPATH（现代）

4、系统默认的位置（和发行版有关）

我的 Ubuntu24 上，/lib64 和 /usr/lib64 这里只有唯一一个 ld-linux-x86-64.so.2，存放了一个动态链接器

Ubuntu 系是在 /lib/x86_64-linux-gnu 或 /usr/lib/x86_64-linux-gnu

（RedHat 才会是 /lib64 和 /usr/lib64 下面一大堆）

注：不会像 Windows 那样，加载 可执行文件同目录/当前工作的目录，除非程序内代码动态加载/编译时用`$ORIGIN`制定可执行文件所在目录

原因：防止你在 /tmp 下执行 ls，如果那个地方恰好有一个同名的假链接库，你就会被劫持

### /usr merge

现代 Linux 新增的点

/lib      系统启动早期必须用的基础库

/usr/lib  普通用户态程序库

/bin      启动必需命令

/usr/bin  普通命令

现在很多发行版已经把它们合并了，即 /bin 和 /usr/bin 类似

### SONAME

用于动态库版本管理

以 git 为例

```
ldd git
        linux-vdso.so.1 (0x0000782999c2e000)
        libpcre2-8.so.0 => /lib/x86_64-linux-gnu/libpcre2-8.so.0 (0x0000782999766000)
        libz.so.1 => /lib/x86_64-linux-gnu/libz.so.1 (0x000078299974a000)
        libc.so.6 => /lib/x86_64-linux-gnu/libc.so.6 (0x0000782999400000)
        /lib64/ld-linux-x86-64.so.2 (0x0000782999c30000)
```

1、它把它的动态链接器本身写死了，绝对路径，ld-linux-x86-64.so.2，用`readelf -l /usr/bin/git` 可显示其链接器：[Requesting program interpreter: /lib64/ld-linux-x86-64.so.2]

动态链接器是加载其他库的先决条件，解决鸡生蛋、蛋生鸡的问题

动态链接器不是普通 NEEDED 依赖，而是 ELF 头里的 INTERP 指定的程序解释器

Windows 则没有这种显示依赖，一切依靠系统自动完成

2、linux-vdso.so.1，内核映射到进程内存中的一个虚拟动态库，不是磁盘上的文件

3、剩下的三个库，都是写到了大版本的 SONAME，没有写路径，右边的 => 指明了具体会在哪个位置寻找

普通库不写死，是为了可移植性

SONAME 本质是一个在 .so 内部的字符串，不是文件名（文件名是完全的版本号）

使得不同主版本号的库像 ls 和 ll 两个不同名字的文件一样，和平共处于同一目录

libgflags.so.2.2 和 libgflags.so.2.3 和 libgflags.so.3（以及对应的真实文件）同时放在 /usr/lib 里，完全没问题

（1）链接名（Linker Name）无后缀，libgflag.so，仅供编译时使用

（2）SONAME，嵌入在库内部，运行时的核心依据

（3）真实文件名，带有完整细粒度版本号

SONAME 相同，但实现版本不同，就不能简单放在同一目录，通常只能让符号链接指向其中一个，另一个版本仍然可以保留，但不能同时通过同一个 SONAME 链接被自动选择

### -dev

不带 -dev = 运行

带 -dev = 编译工具

（1）头文件 /usr/include/

（2）无版本号的软链接 libxxx.so 一个软链接，指向当前最新的带版本号的库文件、

（3）元数据 .pc 和 静态库 .a 

libz.so（无版本号）在同一系统上只能存在一份，且必须指向唯一的默认版本，裸名只能属于某一个版本

所以：lib1 和 lib2 的运行包不冲突，但是 -dev 包都没版本号
