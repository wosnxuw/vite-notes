### 传统开发模式

分许需求 -> 设计 -> 编码 -> 测试

测试目标为发现缺陷。如果难以测试/发现不合理，开发已经完成，为时已晚

### TDD 的出现

2002 年，一个叫做 TDD 的思路被提出来，目标是，测试不再只是验证工具，而成为设计反馈工具

测试 -> 实现 -> 重构

那么到底应该怎么测试？产生了 Chicago 和 London TDD 方向

备注：London 和 Chicago 描述测试风格，和单元测试/集成测试/端到端测试是独立的

### Chicago School

即经典 TDD，解决如何一步一步的实现正确的程序

（1）从小的地方开始，逐步扩展到整个系统

（2）测试可观察到结果（状态验证：返回值、输出、状态等）

（3）尽量使用真实对象/协作者，在高代价位置替换，测试一组对象

### London School

一个对象的主要职责经常不是计算某个返回值，而是协调其他对象，只检查最终状态，可能不能清楚表达协调者的责任

Mock 不只是加速测试的技术，它还可以用来描述：被测对象希望周围存在哪些角色，以及它准备向这些角色发送什么消息。

（1）行为验证：调用顺序是否合理，是否有异常调用、谁调用了谁

（2）将协作者之间用 Mock 隔离，测试单个对象

例子：结账服务需要一个“收款者”，先发现了一个角色，随后再实现真实支付网关

### 对比

Chicago：对象主要体现状态和计算

核心问题：运行后的业务结果对不对？

适合：算法、值对象、领域模型、数据转换、规则计算

London：对象主要体现角色和消息

核心问题：这个对象是否正确履行了自己的协作责任？

适合：工作流编排、应用服务、Controller、外部系统集成、事件发布、消息驱动系统、多个边界系统之间的协调

### 测试问题最终会发展成架构问题

测试一个订单规则时，必须：
```
启动数据库
设置数据库表
启动网络服务
准备框架容器
清理测试数据
```

这说明依赖关系错乱了

### Hexagonal Architecture 六边形架构

实际上和 6 没关系，只是为了避免人们继续把系统想象成上下分层

指：端口与适配器架构、Ports and Adapters

解决问题：业务应用、UI、测试、数据库之间缠绕，必须一起跑

希望：业务应用可以被多种外部驱动方式调用

说白了，没有什么事情是增加中间层解决不了的

只需要在业务和数据库之间，添加 Ports 作为接口，给它们隔开就行了

即实现：

```
 REST ───────┐
 CLI ────────┼──> 输入 Port ──> 应用核心
 Test ───────┘

 应用核心 ──> 输出 Port ──┬──> MySQL
                        ├──> In-memory Fake
                        └──> PostgreSQL
```

London TDD 写测试时会自然产生 Port（需要的角色）

### Onion Architecture 洋葱架构

传统分层的主要问题是：每一层都依赖下层，最终很多代码直接或间接依赖基础设施。

洋葱架构将领域模型放在中心，把基础设施移到外围，所有依赖朝内。

### Clean Architecture

2017 年，将相似思想统一整理

目标：将高级业务策略与易变的技术细节分离

核心：Dependency Rule，源代码依赖只能指向内层

典型四层：

Entities：最核心的企业业务规则

Use Cases：业务流程

Interface Adapters：做格式转换

Frameworks & Drivers：框架/UI/SDK 等

### XP

TDD 是极限编程 12 个核心思想之一

但是用了 TDD 不等于会敏捷

使用方式不当时，TDD 会变成负担，尤其是过度使用 London 风格 Mock

* 为每个 getter、setter 写测试
* 重构一次就要改几十个 Mock
* 测试代码比生产代码复杂很多
* 为了测试而制造无意义接口
* 把“测试先行”变成机械仪式
* 对临时原型也要求完整测试体系
