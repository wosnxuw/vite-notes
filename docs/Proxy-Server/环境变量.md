### 命令行代理

在非 TUN 模式，cmd/bash 里无法 curl google，即无法代理，但是 powershell 是可以的，它可以正常走代理，因为 cmd 的 curl 是 system32 下面的 curl.exe，而 powershell 是集成到 .NET 里面了

有一种临时的办法，不开启 TUN，则是设置临时环境变量

```cmd
set http_proxy=http://127.0.0.1:10808
set https_proxy=http://127.0.0.1:10808
set all_proxy=socks5://127.0.0.1:10808
```

```bash
export http_proxy=http://127.0.0.1:10808
export https_proxy=http://127.0.0.1:10808
export all_proxy=socks5://127.0.0.1:10808
```

这个东西，是否大小写都写，AI 众说纷纭。因为小写的，是 libcurl 的标准；大写的可能一些 Java 在用

一个程序能否走代理，你要排查两个事情：

1、该程序内部调用网络接口的方式

大体有三种：

（1）系统代理。会自动读取设置里配置的代理端口，比如浏览器，各种 GUI 程序。

（2）环境变量。一般是来源于 Linux 的程序，只认 http_proxy 和 https_proxy 环境变量，Windows 不自主定义这两个，所以我们需要重新定义它，骗过这些程序。比如 curl，git，npm。

（3）无法代理。写的比较粗暴，写死了。

所以，我们在 cmd 里配置临时环境变量，后续启动的进程都是子进程，自动继承环境变量，所以 git curl 可用。

2、最后的程序，是否真能拿到 http_proxy 变量

例子 1：我在 Mac 上用 Codex App，它总是出现“正在重新连接”

它的原因是，Codex App 启动了 Codex CLI，而 Codex App 是 GUI 程序，在 Dock 上启动，由 launchd 启动，而 launchd 没有环境变量。因此 CLI 无法继承到 http_proxy

systemd 也一样有这个问题（其实，不依赖用户设置的变量，是 systemd 服务比较稳定的原因之一）

它这里也有用户和系统 launchd 之分，系统确实是开机就在了，但是用户的，是你登录之后才在。你的 GUI，只继承你用户级 launchd。

LaunchAgents 是一个用户级启动任务，用于登录桌面时，运行脚本

但是这样做还是范围有些过大，可以只给 Codex 做一个包装脚本来启动

例子 2:我在配置 openclaw 的时候，发现它怎么也无法代理到 telegram，我就检查了一下，它是这样的 shell 和 systemd 是两套独立的环境变量系统，git 是 shell 的子进程，但是像 openclaw 这种后台进程，是由用户态的 systemd 来唤起的

这导致你可能需要在 systemd 里设置变量，然而这样做以后，我却没有任何效果

我发现根本原因是 openclaw 用到是 nodejs 的 undici 库

```js
const { fetch } = require('undici'); // 或者注释掉这行，直接用 fetch，都一样
fetch('https://api.telegram.org').then(res => console.log('Success:', res.status)).catch(err => console.log('Failed:', err.message));
```

即便是能在命令行里能 curl，我们这行代码也无法执行

根本原因在于：环境变量 http_proxy 只是一个“约定”，而不是系统级的“强制规则”

这种当时唯一的有效途径就是 proxychains

这个工具是一个工具，而不是后台进程。它不虚拟网卡，而是运行时改一个库，然后让后续的程序实现代理

配置文件把默认的 socks4 删掉，然后换成 http 即可

这样 node 和 openclaw 都能代理成功

例子 3：Codex 能对话，但是 AI 运行脚本是无法访问 Google 了

原因是：

shell 分为交互式、非交互式；登录 shell，非登录 shell

两两组合一共是四种，各自独立

交互式/非交互式不用说了

是否登录，决定了是否会“按登录会话初始化”

但是常见只有三种：

1、登录 + 交互：ssh 登录

2、非登录 + 交互：shell 里再次运行其他 shell

3、非登录 + 非交互：执行脚本，ssh 直接执行命令

备注：GUI 的终端模拟器，登录状态需要检查。我这里，在 VSCode 里起终端，就不是登录，在 Ghostty 里起，就是登录。取决于 GUI 程序自己如何决定

检查：

`echo $0` 如果开头有 `-`，则登录，如果没有，则非登录

`echo $- | rg i` 检查输出结果是否包含“i”这个字母，i 表示 interactive

这里以 zsh 为例，它的启动顺序：

```txt
1. /etc/zshenv
2. $ZDOTDIR/.zshenv

如果是登录 Shell：
3. /etc/zprofile
4. $ZDOTDIR/.zprofile

如果是交互式 Shell：
5. /etc/zshrc
6. $ZDOTDIR/.zshrc

如果是登录 Shell：
7. /etc/zlogin
8. $ZDOTDIR/.zlogin
```

所以现在 AI 执行的问题是：在非交互式 shell 里，拿不到 .zshrc 指定的变量，所以需要写一个 .env，source 到 zprofile 和 zshrc 里，并注意幂等性

`config.fish`和`conf.d/*.fish`
